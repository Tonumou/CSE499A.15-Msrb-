### Weekly Update: Abrar Mohammed Tanzim Alam

My primary objective this week was to architect and construct a highly robust, end-to-end data curation pipeline for our Vision Language Model (VLM). The goal was to programmatically ingest, normalize, and process raw, unstructured disaster imagery into a highly balanced, tactical instruction-response dataset ready for QLoRA fine-tuning.

**Work Done:**

1. **Datasets' Exploratory Data Analysis (EDA):**
   Developed comprehensive Jupyter notebooks (`xview_eda.ipynb`, `aider_eda.ipynb`) to programmatically analyze the raw dataset structures. Extracted critical statistical distributions regarding image resolutions, disaster categories, and damage severity levels. This foundational analysis exposed severe class imbalances (e.g., massive overrepresentation of 'no-damage' classes) which directly guided the architectural design of the downstream standardization and prompt augmentation pipelines.

2. **Image Standardization Pipeline:**
   Built robust pre-processing scripts (`support/image_standardizer.py` and `support/aider_image_standardizer.py`) using the `PIL` library to optimize massive raw satellite and aerial imagery. The pipeline intelligently enforces strict RGB color spaces, automatically detects and discards naturally grayscale images (which negatively impact the VLM's multi-modal performance), and caps maximum resolutions at 1280px to prevent VRAM exhaustion (OOM errors) during Qwen2-VL training on T4 GPUs. It also filters out images with edge dimensions below 256px and tracks execution state to allow resume-safe processing. This outputs standardized `.jpg` images and `.json` metadata labels into their corresponding processed directories.

3. **Instruction-Response Ground Truth Generation:**
   Engineered the generation engine (`support/generate_ground_truth.py` and `support/aider_generate_ground_truth.py`) leveraging the `google.genai` SDK and using gemini 3.5 and 2.5 flash models. To overcome the extreme class imbalances identified in EDA, I implemented dynamic prompt multipliers to aggressively over-sample rare disaster events. This successfully yielded a highly balanced week 1 pilot batch of ~500 pairs (496 to be exact) formatted for tactical rescue guidance, which is directly outputted as the raw baseline file **`dataset/train_dataset.jsonl`**.

4. **Reviewer Validation UI Tool:**
   Designed a local graphical user interface (`support/dataset_validator.py`) using `Tkinter` to facilitate rapid human-in-the-loop verification. The tool ingests the generated **`dataset/train_dataset.jsonl`**, rendering the disaster image alongside its synthesized response, and allows team members to Accept, Reject, or Undo evaluations based on a strict Rescue Actionability Rubric (RAR). This rubric ensures the text accurately identifies structural collapses, maps correct geospatial quadrants, and provides hallucination-free hazard avoidance tips, outputting only the highest quality samples into **`dataset/verified_dataset.jsonl`**.

5. **Dataset Sanitization Engine:**
   Created an automated sanitization script (`support/sanitize_dataset.py`) utilizing regular expressions to rigorously clean both newly generated and legacy xView datasets. The engine actively scrubs first-person conversational Ai language (e.g., "I apologize," "As an AI" etc) and strips out generic placeholder brackets generated by the LLM. It enforces strict length boundaries by aggressively discarding any responses under 80 words (which typically indicate truncated or insufficiently detailed outputs) or over 600 words (which indicate hallucinatory or overly verbose conversational filler), ensuring only concise, highly actionable tactical guidance is retained. Furthermore, it utilizes a round-robin injection technique to randomly replace repetitive, default system prompts with a diverse array of 6 distinct tactical instruction variants, ensuring the VLM learns generalized query comprehension. This outputs a completely scrubbed, diverse, and randomized master file as **`dataset/final_training_dataset.jsonl`**.

6. **Final Verified Dataset Statistics Visualization:**
   Developed `support/dataset_stats.py` to automatically parse the finetune-ready **`dataset/final_training_dataset.jsonl`** and generate comprehensive visual reports using `matplotlib` and `seaborn` libraries. This tool plots the post-generation distributions of disaster types and damage severities, providing immediate, verifiable proof that our balancing mechanisms successfully mitigated the raw dataset's intrinsic biases.

7. **Dataset Train/Test/Validation Splitting:**
   Implemented a rigorous data partitioning script (`support/dataset_split.py`) to systematically divide the finetune-ready **`dataset/final_training_dataset.jsonl`** into strict training, validation, and testing splits (outputting **`dataset/train_split.jsonl`**, **`dataset/val_split.jsonl`**, and **`dataset/test_split.jsonl`**). This prepares the small sample pool for seamless ingestion by the Unsloth QLoRA training framework for the proof of concept finetuning of the Qwen2-VL model.

8. **Cross-Dataset Pipeline Adaptation (AIDER Transfer):**
   Successfully adapted the entire xView-centric data curation architecture to seamlessly ingest and process the AIDER aerial dataset. This required extensive refactoring of the standardizer and generation scripts to dynamically interpret AIDER's unique class-based directory structure and normalize differing coordinate systems without disrupting the core pipeline logic. SO that the AIDer pipeline can utilize the downstream processing scripts of the Xview dataset.

9. **Synthetic Metadata Synthesization:**
   Designed an advanced bridging script (`support/aider_synthesize_metadata.py`) specifically to solve AIDER's lack of precise ground-truth annotations. The script queries Gemini to synthetically generate xBD-equivalent metadata (bounding polygon approximations and structural severity classifications) and outputs them as temporary `.json` files, flawlessly normalizing the diverse AIDER data into our strict uniform training schema.
